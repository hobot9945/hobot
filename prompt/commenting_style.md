# Инструкция по оформлению кода на Rust

При генерации или рефакторинге кода строго придерживайся следующего стиля документации и комментариев (язык 
комментариев — Русский).

## 1. Модули (mod)
Используй блочные документирующие комментарии //! в начале файла.

Структура:
- Заголовок, должен отражать назначения модуля.
- Описание модуля. Цели - облегчение понимания.
- Более формальное описание модуля, например список под подзаголовком ОТВЕТСТВЕННОСТЬ. Цель - увеличение информативности.
- Прочие разделы с заголовками уровня # по необходимости.

## 2. Структуры и Перечисления
Используй /// для документирования типа.

- Описание назначения структуры.
- Для полей структуры используй встроенные комментарии // для кратких пояснений (тип, назначение), особенно если назначение неочевидно из имени.

## 3. Функции и Методы
Используй /// для документирования функций.

- Описание: Краткое описание действия функции.
- Алгоритм работы: (опционально) пошаговое описание логики, если она сложная (используй маркированный список).
- Секции:
  Используй заголовки уровня # для основных категорий: # Паника, # Ошибки (или # Errors), # Возвращаемое 
  значение (или # Returns), # Побочные эффекты (или # Side effects).
  Используй заголовки уровня ## для детализации параметров или частных случаев: ## Параметры, ## Поведение при ошибках.
- Форматирование внутри секций:
  Для возвращаемого значения: # Возвращаемое значение
  Тип: &AppConfig: Ссылка на конфигурацию.
- Если есть побочные эффекты, они должны быть явно описаны. Например, изменение параметра, передаваемого по мутабельной 
  ссылке, доступ к нелокальным переменным по чтению/записи.

## 4. Внутренние комментарии
Используй // для комментариев внутри кода.

- Ставь их над строкой кода или справа от нее.
- Комментарии должны объяснять смысловые блоки кода, даже если они описаны в док-комментариях.
- Не оставляй без комментариев неочевидные моменты, обходы особенностей платформы.

## 5. Закрывающие скобки

Для облегчения навигации и визуального разграничения блоков кода обязательно ставь комментарии после закрывающих 
фигурных скобок.

- **Обязательны для:** модулей (`mod`), структур (`struct`), перечислений (`enum`), блоков реализации (`impl`), 
- функций (`fn`), управляющих структур (`if`, `loop`, `match`...).
- **Желательны для:** крупных циклов, блоков `match` и `if` (если тело занимает более 5-10 строк).

- **Формат:**
    - Для типов и модулей: `}   // TypeName`
    - Для функций: `}   // function_name()` (скобки обязательны для отличия от типа)
    - Для impl блоков: `}   // impl TypeName
    - Для управляющих структур `}    // if`, `}    // loop`, `}    // match`, ...
  
## 6. Разное
Не включай в комментарии ссылки на твои источники, к примеру, "[web:123]".

## 7. Общие требования
- Стиль: Технический, лаконичный, без "воды".
- Язык: Русский.
