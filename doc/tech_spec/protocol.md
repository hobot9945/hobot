# Протокол взаимодействия (Protocol Specification)

Документ описывает форматы обмена данными между компонентами системы: AI (Web), Расширение (Extension) и Агент (Hobot).

---

## 0. Термины и скобки

В протоколе используются текстовые маркеры (скобки) для однозначного выделения сообщений в текстовом потоке:

- `<<<ai ... >>>ai` — директивы от AI к Хоботу (через расширение в stdin).
- `<<<ext ... >>>ext` — служебные сообщения расширения к Хоботу (в stdin).
- `<<<hbt ... >>>hbt` — сообщения Хобота:
    - в stdout → расширению (управляющие/служебные),
    - в сторону AI через поле ввода сайта (отчёты по директивам, ошибки).

---

## 0.1. Транспорт Native Messaging (обёртка)

И stdin, и stdout Hobot используют Chrome Native Messaging формат:

- 4 байта длины (native endian, `u32`)
- далее JSON‑тело сообщения

Фактическая полезная нагрузка всегда упакована в JSON‑объект вида:

```json
{
  "text": "СТРОКА_ПРОТОКОЛА"
}
```

Где `СТРОКА_ПРОТОКОЛА` — это один (или несколько) протокольных пакетов в маркерах `<<<ai/<<<ext/<<<hbt ...`.

## 1. Поток AI → Хобот (Директивы)
Маршрут: AI (Web Page) → Extension (message) → Hobot (stdin)

AI передает агенту пакет директив на исполнение. Пакет всегда заключен в `<<<ai ... >>>ai`.

### 1.1 Формат директивы

```text
<<<ai DIRECTIVE_NUM SESSION_ID
    { 
        JSON_PAYLOAD 
    }
>>>ai DIRECTIVE_NUM SESSION_ID
```

- `DIRECTIVE_NUM`: целое число (1, 2, ...), строго возрастающее на 1. Это номер директивы внутри сессии.
  Новая директива должна появляться только после завершения предыдущей (по смыслу протокола).
- `SESSION_ID`: уникальный шестисимвольный идентификатор сессии (шестнадцатиричное число, заглавные буквы).
  Генерируется расширением и передается агенту в пакете инициализации (`INIT_SESSION`).
  Агент переправляет его AI, и далее расширение будет принимать от AI директивы только с таким идентификатором сессии.

Обработка невалидных директив:
- Не соблюден формат тегов → агент формирует отчёт об ошибке и отправляет его AI.
- Идентификатор сессии не равен текущему → агент формирует отчёт об ошибке и отправляет его AI.
- Номер директивы меньше или равен номеру последней исполненной → агент формирует отчёт об ошибке и отправляет его AI.
- Данные тега `<<<ai` не соответствуют данным тега `>>>ai` → агент формирует отчёт об ошибке.
- Пустое тело директивы → агент формирует отчёт об ошибке.

Важно:
- Агент **не является источником истины** по “последнему номеру директивы”: контроль очереди реализован на стороне расширения.
- Если расширение по ошибке повторно пришлёт старую директиву в stdin, агент выполнит её повторно.

### 1.2 JSON директивы

```json
{
    "dir_comment": "Опциональный комментарий директивы",
    "commands": [
        {
            "cmd_comment": "Опциональный комментарий команды",
            "cmd_id": 1,
            "name": "имя_команды",
            "params": ["параметр 1", "параметр 2"],
            "allowed_timeout": 50000
        }
    ]
}
```

Поля:
- `dir_comment`: (опционально) комментарий AI.
- `commands`: массив команд.
- `cmd_comment`: (опционально) комментарий к конкретной команде.
- `cmd_id`: идентификатор команды внутри директивы (начинается с 1, +1).
- `name`: имя команды.
- `params`: (опционально) массив строк; может отсутствовать, если команда без параметров.
- `allowed_timeout`: (опционально) максимально допустимое время исполнения (мс), если не устраивает заданное в файле настроек Хобота.

## 2. Поток Расширение → Хобот (Служебный)

Маршрут: Extension → Hobot (stdin)

Расширение отправляет служебные пакеты в скобках `<<<ext ... >>>ext`.

### 2.1 Формат

```text
<<<ext
    {
        JSON_PAYLOAD
    }
>>>ext
```

### 2.2 Инициализация агента (INIT_SESSION)

Отправляется сразу после установления постоянного соединения с процессом агента.

```json
{
  "type": "INIT_SESSION",
  "payload": {
    "session_id": "A1B2C3",
    "browser": "chrome",
    "ai_url": "https://chat.deepseek.com",
    "window_title": "https://chat.deepseek.com [A1B2C3]",
    "os_readonly": true,
    "step_through": true
  }
}
```

Поля `payload`:

- `session_id`: уникальный идентификатор сессии, строка из шести шестнадцатиричных символов (заглавные буквы).
  Генерируется расширением.
- `browser`: идентификатор браузера (`chrome`, `firefox`, `edge`).
- `ai_url`: URL origin страницы AI.
- `window_title`: заголовок вкладки, который расширение замораживает. Строится как `ai_url + " [" + session_id + "]"`.
- `os_readonly`: начальное значение флага ограничения записи (true — только безопасные команды, false — полный доступ).
- `step_through`: начальное значение флага пошагового режима (true — каждая директива требует подтверждения пользователя, false — автоматическое исполнение).

### 2.3 Завершение работы (COMPLETION)

Перед завершением работы расширение посылает Хоботу сообщение закрытия, чтобы он мог легально завершить свою
деятельность.

```json
{
  "type": "COMPLETION"
}
```

### 2.4 Ошибки протокола (PROTOCOL_ERROR)

Если расширение обнаруживает нарушение протокола (например, `DIRECTIVE_NUM` не тот, невалидный JSON, нарушена очередность),
оно отправляет служебное сообщение агенту. Агент, действуя как прозрачный прокси, пересылает текст ошибки в чат ИИ.

```json
{
  "type": "PROTOCOL_ERROR",
  "error_message": "Текст ошибки для ИИ"
}
```

*   `type`: Константа `"PROTOCOL_ERROR"`.
*   `error_message`: Строка с описанием проблемы, понятная для LLM.

**Поведение агента:**
При получении такого пакета агент формирует отчёт и отправляет содержимое `error_message` в поле ввода чата,
обернув его в транспортные маркеры `<<<hbt SESSION_ID ... >>>hbt SESSION_ID` (см. раздел 4.5).

**Типичные ошибки (генерируются текстом в `error_message`):**
- Пропущен номер директивы (DIRECTIVE_NUM gap).
- Нарушен формат `<<<ai ... >>>ai`.
- Таймаут ожидания закрывающего тега директивы.

### 2.5 Изменение флага пошагового режима (CHANGE_STEP_THROUGH)

Расширение посылает агенту обновлённое значение флага `step_through` при изменении
кнопки управления исполнением в popup UI.

```json
{
  "type": "CHANGE_STEP_THROUGH",
  "value": true
}
```

*   `type`: Константа `"CHANGE_STEP_THROUGH"`.
*   `value`: `true` — пошаговый режим (паузы перед каждой директивой), `false` — автоматический режим.

**Поведение агента:**
Обновляет внутренний флаг `step_through` в контексте сессии. При `step_through = true` перед исполнением каждой
директивы агент показывает модальное окно с запросом подтверждения.

Агент **не** отправляет `DIRECTIVE_COMPLETED` в ответ на это сообщение.

### 2.6 Изменение флага ограничения записи (CHANGE_OS_READONLY)

Расширение посылает агенту обновлённое значение флага `os_readonly` при изменении
кнопки уровня доступа в popup UI.

```json
{
  "type": "CHANGE_OS_READONLY",
  "value": false
}
```

*   `type`: Константа `"CHANGE_OS_READONLY"`.
*   `value`: `true` — только безопасные команды (белый список), остальные требуют подтверждения пользователя.
    `false` — все команды исполняются без ограничений.

**Поведение агента:**
Обновляет внутренний флаг `os_readonly` в контексте сессии.

Агент **не** отправляет `DIRECTIVE_COMPLETED` в ответ на это сообщение.

## 3. Поток Хобот → Расширение (Управляющий stdout)

Маршрут: **Hobot stdout → Native Messaging port → background.js → content scripts**

Используется для:

- сброса флага занятости (разрешение на следующую директиву),

### 3.1 Формат

```text
<<<hbt
    {
        JSON_PAYLOAD
    }
>>>hbt
```

Специально отметим, что сообщения вида `<<<hbt DIRECTIVE_NUM SESSION_ID ... >>>hbt DIRECTIVE_NUM SESSION_ID` идут от
агента в сторону AI, а для отправки сообщений в сторону расширения используется упрощённая форма `<<<hbt ... >>>hbt`
(без метаданных в тегах).

### 3.2 Сигнал готовности (DIRECTIVE_COMPLETED)

Сообщает, что агент завершил обработку и готов к следующему сообщению.

```json
{
  "type": "DIRECTIVE_COMPLETED"
}
```

**Момент завершения (Completion Point).**

Агент обязан отправлять `DIRECTIVE_COMPLETED` **только после успешной публикации отчёта в AI**: отчёт в формате
`<<<hbt DIRECTIVE_NUM SESSION_ID ... >>>hbt DIRECTIVE_NUM SESSION_ID` полностью введён в поле ввода и выполнено действие
отправки сообщения (нажатие Enter).

Расширение использует `DIRECTIVE_COMPLETED` как единственный сигнал для сброса `isAgentBusy` и разрешения передачи
следующей директивы в `stdin`. Флаг `isAgentBusy` используется расширением для захвата фокуса полем ввода чата и 
блокировки перехода на другую вкладку браузера. Это нужно чтобы агент мог поместить в поле ввода сообщение.

**Когда отправляется:**
- После обработки AI-директивы (успешной или аварийной).
- После обработки пакета инициализации `INIT_SESSION`.

**Когда не отправляется:**
- При получении `PROTOCOL_ERROR`.
- При получении `COMPLETION`.
- При получении `CHANGE_STEP_THROUGH` или `CHANGE_OS_READONLY`.

## 4. Поток Хобот → AI (Отчеты)

**Маршрут:** Hobot → AI (ввод текста в чат).

Агент формирует отчет в формате **Markdown**. Отчет всегда оборачивается в транспортные теги, чтобы AI мог поставить
его в соответствие директиве, но внутри тегов находится человекочитаемый текст.

Для ответа на директивы AI используются теги вида `<<<hbt DIRECTIVE_NUM SESSION_ID ... >>>hbt DIRECTIVE_NUM SESSION_ID`.
Для остальных сообщений (инициализация, ошибки расширения, завершение) — теги вида `<<<hbt SESSION_ID ... >>>hbt SESSION_ID`.

### 4.1. Принцип "Адаптивного забора" (Adaptive Fences)

Для вывода результатов команд (stdout/stderr) агент использует блоки кода Markdown.
Чтобы избежать поломки разметки, если вывод команды сам содержит тройные обратные кавычки (` ``` `), агент динамически
выбирает длину "забора":

1. Агент сканирует текст результата.
2. Находит самую длинную непрерывную цепочку символов `` ` `` (длиной *N*).
3. Оборачивает результат в забор из `N + 1` (но не менее 3) обратных кавычек.

### 4.2. Структура успешного/частичного отчета

``````markdown
`<<<hbt DIRECTIVE_NUM SESSION_ID`
# 🏁 Отчет по директиве #DIRECTIVE_NUM
> **Статус**: ✅ Выполнено (2/2)

### 1. `shell_cmd` (ID: 1)
- **Результат**: ✅ OK
- **Вывод**:

```
содержимое stdout
```

### 2. `read_file` (ID: 2)
- **Результат**: ✅ OK
- **Вывод**:

```
содержимое файла
```
`>>>hbt DIRECTIVE_NUM SESSION_ID`
``````

**Легенда:**
*   `# 🏁 ...` — Заголовок уровня H1. Четко разделяет ответы агента в истории чата.
*   `> **Статус**: ...` — Сводная статистика (Completed/Total). `✅` при полном успехе, `⚠️` при частичном.
*   `### N. ...` — Заголовок уровня H3 для каждой команды. Содержит имя и ID.
*   Если директива прервана ошибкой, перед секциями команд выводится блок **Детали директивы** с причиной остановки.

### 4.3. Структура отчета при ошибке директивы

Если директива не может быть выполнена полностью (ошибка парсинга, неизвестная команда, ошибка хэндлера),
агент формирует отчёт со статусом `⚠️` и указанием причины остановки:

``````markdown
`<<<hbt DIRECTIVE_NUM SESSION_ID`
# 🏁 Отчет по директиве #DIRECTIVE_NUM
> **Статус**: ⚠️ Выполнено (1/2)

**Детали директивы**:

```
Выполнение директивы остановлено: команда cmd_id=2 завершилась ошибкой.
```

### 1. `shell_cmd` (ID: 1)
- **Результат**: ✅ OK
- **Вывод**:

```
OK
```

### 2. `unknown_cmd` (ID: 2)
- **Результат**: ⚠️ ОШИБКА
- **Текст ошибки**:

```
Неизвестная команда: 'unknown_cmd'
```
`>>>hbt DIRECTIVE_NUM SESSION_ID`
``````

### 4.4. Отчет об инициализации (Приветствие)

Сразу после получения пакета `INIT_SESSION` от расширения (см. п. 2.2), агент отправляет в чат
приветственное сообщение. Это позволяет ИИ узнать, в каком окружении он работает.

``````markdown
`<<<hbt A1B2C3`
# 🚀 Хобот готов к работе.

**Контекст сессии:**
```
session_id: "A1B2C3"
next_directive_num: 1
browser: "chrome"
ai_url: "https://chat.deepseek.com"
window_title: "https://chat.deepseek.com [A1B2C3]"
```
`>>>hbt A1B2C3`
``````

**Особенности:**
* Блок "Контекст сессии" заполняется данными из payload `INIT_SESSION`.
* Поле `next_directive_num` указывает AI, какой номер использовать для первой директивы.

### 4.5. Ошибка протокола расширения (PROTOCOL_ERROR)

Используется, когда расширение обнаружило нарушение протокола (см. п. 2.4) и уведомило об этом агента.
Агент действует как прозрачный прокси: публикует ошибку в чат AI, попутно помещая сообщение в свой журнал ошибок.

``````markdown
`<<<hbt SESSION_ID`
# ⛔ Ошибка протокола расширения
**Детали**:
```
DIRECTIVE_NUM has gap. expected=5 received=7
```
`>>>hbt SESSION_ID`
``````

### 4.6. Завершение работы агента (COMPLETION)

Перед остановкой (после получения пакета расширения `COMPLETION`, см. п. 2.3) агент публикует в AI финальное сообщение.

```markdown
`<<<hbt SESSION_ID`
# 📴 Хобот завершает работу по запросу расширения.
`>>>hbt SESSION_ID`
```

## 5. Правила совместимости и устойчивости

### 5.1 Строгая последовательность директив

Расширение ведёт внутренний счётчик `_lastDirectiveNum`, обновляя его по сигналу `DIRECTIVE_COMPLETED`.

Правило приёма директив AI: расширение передаёт агенту **только** директиву с `DIRECTIVE_NUM == _lastDirectiveNum + 1`.

Все директивы с `DIRECTIVE_NUM <= _lastDirectiveNum` (дубликаты/устаревшие) 
**вызывают генерацию сообщения об ошибке протокола**.

- `DIRECTIVE_NUM > _lastDirectiveNum + 1`: обнаружен разрыв последовательности (пропущена ожидаемая директива).
  Расширение обязано:
    1) отправить агенту служебную ошибку `<<<ext ... >>>ext` с `"type": "PROTOCOL_ERROR"` и
       `"error_message": "DIRECTIVE_NUM has gap."` (см. п. 2.4),
       чтобы агент пробросил её в AI;
    2) не передавать агенту директивы с номерами больше ожидаемого до тех пор, пока не будет получена директива
       с `DIRECTIVE_NUM == _lastDirectiveNum + 1`.

### 5.2 Текстовый транспорт

- Все пакеты являются текстом и должны корректно выделяться по маркерам `<<<...` / `>>>...`.
- Внутри JSON допускаются обычные переводы строк/пробелы.

### 5.3 Разделение каналов

- stdin: директивы AI и служебные пакеты расширения.
- stdout: управляющие и критические сообщения агента расширению (Native Messaging формат: 4 байта длины + JSON).
- stderr: лог агента (перенаправляется в файл bat-скриптом запуска).
- поле ввода AI: отчёты агента по директивам (эмуляция клавиатурного ввода через clipboard + Ctrl+V).
