# Протокол взаимодействия (Protocol Specification)

Документ описывает форматы обмена данными между компонентами системы: AI (Web), Расширение (Extension) и Агент (Hobot).

---

## 0. Термины и скобки

В протоколе используются текстовые маркеры (скобки) для однозначного выделения сообщений в текстовом потоке:

- `<<<ai ... >>>ai` — директивы от AI к Хоботу (через расширение в stdin).
- `<<<ext ... >>>ext` — служебные сообщения расширения к Хоботу (в stdin).
- `<<<hbt ... >>>hbt` — сообщения Хобота:
    - в stdout → расширению (управляющие/служебные),
    - в сторону AI через поле ввода сайта (отчёты по директивам, некритические и критические ошибки).

---

## 1. Поток AI → Хобот (Директивы)
Маршрут: AI (Web Page) → Extension (message) → Hobot (stdin)

AI передает агенту пакет директив на исполнение. Пакет всегда заключен в `<<<ai ... >>>ai`.

### 1.1 Формат директивы

```text
<<<ai DIRECTIVE_NUM SESSION_ID
    { 
        JSON_PAYLOAD 
    }
>>>ai DIRECTIVE_NUM SESSION_ID
```

- `DIRECTIVE_NUM`: целое число (1, 2, ...), строго возрастающее на 1. Это номер директивы внутри сессии.
  Новая директива должна появляться только после завершения предыдущей (по смыслу протокола).
- `SESSION_ID`: уникальный шестисимвольный идентификатор сессии. Он генерируется расширением и передается агенту
  в пакете инициализации ("type": "INIT_SESSION"). Агент переправляет его AI, и далее расширение будет принимать от AI
  директивы только с таким идентификатором сессии. Это делается для повышения устойчивости выделения директив 
  из веб-страницы.
- Директивы молча игнорируются агентом в следующих случаях:
  - не соблюден формат тегов;
  - идентификатор сессии не равен текущему;
  - номер директивы меньше или равен номеру последней исполненной;
  - данные тега <<<ai не соответствуют данным тега >>>ai;


### 1.2 JSON директивы

```json
{
    "dir_comment": "Опциональный комментарий директивы",
    "commands": [
        {
            "cmd_comment": "Опциональный комментарий команды",
            "cmd_id": 1,
            "name": "имя_команды",
            "params": ["параметр 1", "параметр 2"],
            "allowed_timeout": 50000
        }
    ]
}
```

Поля:
- `dir_comment`: (опционально) комментарий AI.
- `commands`: массив команд.
- `cmd_comment`: (опционально) комментарий к конкретной команде.
- `cmd_id`: идентификатор команды внутри директивы (начинается с 1, +1).
- `name`: имя команды.
- `params`: (опционально) массив строк; может отсутствовать, если команда без параметров.
- `allowed_timeout`: (опционально) максимально допустимое время исполнения (мс), если не устраивает заданное в файле настроек Хобота.

## 2. Поток Расширение → Хобот (Служебный)

Маршрут: Extension → Hobot (stdin)

Расширение отправляет служебные пакеты в скобках `<<<ext ... >>>ext`.

### 2.1 Формат

```text
<<<ext
    {
        JSON_PAYLOAD
    }
>>>ext
```


### 2.2 Инициализация агента (INIT_SESSION)

Отправляется сразу после установления постоянного соединения с процессом агента.

```json
{
  "type": "INIT_SESSION",
  "payload": {
    "session_id": "123456",
    "browser": "chrome/firefox/...",
    "ai_url": "https://...",
    "window_title": "название вкладки (замороженное, уникализированное)"
  }
}
```

Поля `payload`:

- `session_id`: уникальный идентификатор сессии, строка, содержащая шестизначное шестнадцатиричное число, заглавные буквы. 
                Генерируется расширением.    
- `browser`: идентификатор браузера
- `ai_url`: URL страницы AI.
- `window_title`: заголовок вкладки, который расширение замораживает. Он строится и ai_url + уникальный суффикс
                  `[HBT-session_id]`).


### 2.3 Завершение работы (COMPLETION)

Перед завершением работы расширение посылает Хоботу сообщение закрытия, чтобы он мог легально завершить свою 
деятельность.


```json
{
  "type": "COMPLETION"
}
```

### 2.4 Ошибки протокола (PROTOCOL_ERROR)

Если расширение обнаруживает нарушение протокола (например, `DIRECTIVE_NUM` не тот, невалидный JSON, нарушена очередность), 
оно отправляет служебное сообщение агенту. Агент, действуя как прозрачный прокси, пересылает текст ошибки в чат ИИ.

**Формат сообщения:**

```
{
  "type": "PROTOCOL_ERROR",
  "error_message": "Текст ошибки для ИИ (например: DIRECTIVE_NUM mismatch: expected 5, received 6)"
}
```

*   `type`: Константа `"PROTOCOL_ERROR"`.
*   `error_message`: Строка с описанием проблемы на английском (или языке промпта), понятная для LLM.

**Поведение агента:**
При получении такого пакета агент формирует отчет типа `EXTENSION_ERROR` и отправляет содержимое `error_message` в поле ввода чата,
обернув его в транспортные маркеры ответа Хобота вида `<<<hbt SESSION_ID ... >>>hbt SESSION_ID` (см. раздел 4.5).
Это позволяет ИИ увидеть системную ошибку транспорта и сгенерировать исправленную директиву.

**Типичные ошибки (генерируются текстом в error_message):**
- Пропущен номер директивы (DIRECTIVE_NUM gap).
- Нарушен формат `<<<ai ... >>>ai`.
- Таймаут ожидания закрывающего тега директивы.

## 3. Поток Хобот → Расширение (Управляющий stdout)

Маршрут: Hobot (stdout) → Extension

Используется для:

- сброса флага занятости (разрешение на следующую директиву),
- сигналов о критических ошибках (чтобы расширение могло принять решение: перезапуск/остановка/alert).


### 3.1 Формат

```text
<<<hbt
    { 
        JSON_PAYLOAD    
    }
>>>hbt
```

### 3.2 Сигнал готовности (DIRECTIVE_COMPLETED)

Сообщает, что агент завершил обработку и готов к следующему сообщению.

**Момент завершения (Completion Point).**  

Агент обязан отправлять `DIRECTIVE_COMPLETED` **только после успешной публикации отчёта в AI**: отчёт в формате 
`<<<hbt DIRECTIVE_NUM SESSION_ID ... >>>hbt DIRECTIVE_NUM SESSION_ID` полностью введён в поле ввода и выполнено действие 
отправки сообщения (нажатие Enter / эквивалент).

Расширение использует `DIRECTIVE_COMPLETED` как единственный сигнал для сброса `isAgentBusy` и разрешения передачи 
следующей директивы в `stdin`.
```json
{
  "type": "DIRECTIVE_COMPLETED"
}
```

**Важно**:
`DIRECTIVE_COMPLETED` отправляется только как сигнал завершения обработки AI-директивы и пакета инициализации от 
расширения `INIT_SESSION`. При получении служебных сообщений расширения (`PROTOCOL_ERROR`, `COMPLETION`) агент не 
отправляет `DIRECTIVE_COMPLETED`.

### 3.3 Критическая ошибка (CRITICAL)

```json
{
  "type": "CRITICAL_ERROR",
  "error_message": "Watchdog timer expired"
}
```
Специально отметим, что сообщения вида `<<<hbt DIRECTIVE_NUM SESSION_ID ... >>>hbt DIRECTIVE_NUM SESSION_ID` идут от
агента в сторону AI, в для отправки сообщений в сторону расширения используется упрощенная форма `<<<hbt ... >>>hbt`.

## 4. Поток Хобот → AI (Отчеты)

**Маршрут:** Hobot → AI (ввод текста в чат).

Агент формирует отчет в формате **Markdown**. Отчет всегда оборачивается в транспортные теги для надежной детекции 
границ сообщения расширением, но внутри тегов находится человекочитаемый текст. Для ответа на директивы AI используются 
теги вида `<<<hbt DIRECTIVE_NUM SESSION_ID ... >>>hbt DIRECTIVE_NUM SESSION_ID`, для остальных запросов `<<<hbt ... >>>hbt`.

### 4.1. Принцип "Адаптивного забора" (Adaptive Fences)

Для вывода результатов команд (stdout/stderr) агент использует блоки кода Markdown.
Чтобы избежать поломки разметки, если вывод команды сам содержит тройные обратные кавычки (` ``` `), агент динамически
выбирает длину "забора":

1. Агент сканирует текст результата.
2. Находит самую длинную непрерывную цепочку символов `` ` `` (длиной *N*).
3. Оборачивает результат в забор из `N + 1` (но не менее 3) обратных кавычек.

### 4.2. Структура успешного/частичного отчета

````markdown
<<<hbt DIRECTIVE_NUM SESSION_ID
# 🏁 Отчет по директиве #DIRECTIVE_NUM
> **Статус**: ✅ Выполнено (2/2)

### 1. `shell_cmd` (ID: 101)
- **Результат**: ✅ OK
- **Вывод**:

### 2. `read_file` (ID: 102)
- **Результат**: ⚠️ ОШИБКА
- **Текст ошибки**:
```
Файл 'C:\secrets.txt' не найден или занят другим процессом.
```
>>>hbt DIRECTIVE_NUM SESSION_ID

````

**Легенда:**
*   `# 🏁 ...` — Заголовок уровня H1. Четко разделяет ответы агента в истории чата.
*   `> ...` — Цитата для сводной статистики (Total/Completed).
*   `### N. ...` — Заголовок уровня H3 для каждой команды. Содержит имя и ID.
*   `✅` / `⚠️` / `❌` — Визуальные индикаторы успеха, ошибки команды или критического сбоя.

### 4.3. Структура отчета при фатальном сбое

Если директива не может быть выполнена (ошибка парсинга, нарушение протокола) или прервана вмешательством пользователя:

````markdown
<<<hbt DIRECTIVE_NUM SESSION_ID
# ⛔ Сбой директивы #DIRECTIVE_NUM
> **Тип ошибки**: CRITICAL / EXTENSION_ERROR / PARSE_ERROR / INTERRUPTED 

**Детали**:
```
Вмешательство пользователя: обнаружено движение мыши.
Выполнение прервано на команде ID: 5.
```
>>>hbt DIRECTIVE_NUM SESSION_ID
````

### 4.4. Отчет об инициализации (Приветствие)

Сразу после запуска и получения пакета инициализации `INIT_SESSION` от расширения (см. п. 2.2), агент отправляет в чат 
приветственное сообщение. Это позволяет ИИ узнать, в каком окружении он работает.

````markdown
<<<hbt E43FC1
# 🚀 Хобот готов к работе.

**Контекст сессии:**
```
session_id: "E43FC1"
browser: "chrome"
ai_url: "https://chat.deepseek.com"
window_name: "https://chat.deepseek.com [E43FC1]"
```
>>>hbt E43FC1
````

**Особенности:**

* Блок "Контекст сессии" заполняется данными, полученными от расширения (payload из `INIT_SESSION`).

### 4.5. Ошибка протокола расширения (PROTOCOL_ERROR)

Используется, когда расширение обнаружило нарушение протокола (см. 2.4 `PROTOCOL_ERROR`) и уведомило об этом агента.
Агент действует как прозрачный прокси: публикует ошибку в чат AI, чтобы LLM могла пересобрать корректную директиву.

**Формат:**
````markdown
<<<hbt SESSION_ID
# ⛔ Ошибка протокола расширения
> **Тип ошибки**: PROTOCOL_ERROR

**Детали**:
DIRECTIVE_NUM has gap. expected=5 received=7

>>>hbt SESSION_ID
````

### 4.6. Завершение работы агента (COMPLETION)
Перед остановкой (после получения пакета расширения COMPLETION, см. 2.3) агент публикует в AI финальное сообщение,
чтобы завершение сессии было явно видно в истории чата.

**Формат:**
```markdown
<<<hbt SESSION_ID
# 📴 Хобот завершает работу
> **Причина**: COMPLETION
>>>hbt SESSION_ID
```

## 5. Правила совместимости и устойчивости

### 5.1 Строгая последовательность директив

Расширение ведёт `last_completed_dir_id`, обновляя его **только** по stdout-сообщениям агента 
`DIRECTIVE_COMPLETED.processed_dir_id`.

Правило приёма директив AI: расширение передаёт агенту **только** директиву с `DIRECTIVE_NUM == last_completed_dir_id + 1`.

Все директивы с `DIRECTIVE_NUM <= last_completed_dir_id` (дубликаты/устаревшие), которые могут повторно попадать в парсер 
из-за промотки/повторного чтения DOM, расширение **игнорирует без генерации сообщений об ошибке**.

- `DIRECTIVE_NUM > last_completed_dir_id + 1`: обнаружен разрыв последовательности (пропущена ожидаемая директива).
  Расширение обязано:
    1) отправить агенту служебную ошибку `<<<ext ... >>>ext` с `"type": "PROTOCOL_ERROR"` и 
       `"error_message": "DIRECTIVE_NUM has gap."` (см. 2.4), 
       чтобы агент пробросил её в AI;
    2) показать пользователю системный `alert` с кратким описанием (expected/received), т.к. это признак сбоя 
       доставки/парсинга;
    3) не передавать агенту директивы с номерами больше ожидаемого до тех пор, пока не будет получена директива 
       с `DIRECTIVE_NUM == last_completed_dir_id + 1`.
       
Примечание (режим отладки): при обнаружении разрыва последовательности (`DIRECTIVE_NUM > last_completed_dir_id + 1`)
       расширение дополнительно показывает пользователю системный `alert` с `expected/received`.
       Поведение может быть изменено/убрано после стабилизации парсинга.

### 5.2 Текстовый транспорт

- Все пакеты являются текстом и должны корректно выделяться по маркерам `<<<...` / `>>>...`.
- Внутри JSON допускаются обычные переводы строк/пробелы.


### 5.3 Разделение каналов

- stdin: директивы AI и служебные пакеты расширения.
- stdout: управляющие и критические сообщения агента расширению.
- поле ввода AI: отчёты агента по директивам (для обратной связи AI).
