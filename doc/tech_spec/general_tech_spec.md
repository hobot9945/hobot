# Техническое задание: агент Хобот

## 1. Назначение системы

Агент **Хобот** предназначен для выполнения команд, формируемых искусственным интеллектом (ИИ) в веб‑интерфейсе, с целью управления хостовым компьютером пользователя: запуск внешних программ, выполнение скриптов, управление окнами, вводом с клавиатуры и мыши, создание скриншотов и другие действия с максимальными правами в системе.

Хобот не принимает решений самостоятельно, а выступает детерминированным исполнителем директив ИИ, обеспечивая при этом строгие гарантии безопасности протокола и предсказуемое поведение при ошибках и вмешательстве пользователя.

## 2. Общая архитектура и роли

- **ИИ (через веб‑страницу)**  
  - Формирует директивы в текстовом виде.  
  - Читает отчёты Хобота из поля ввода веб‑страницы.  
  - При необходимости корректирует своё поведение на основе отчётов.

- **Пользователь**  
  - Общается с ИИ в обычном чате.  
  - Наблюдает за действиями Хобота.  
  - Вмешивается при необходимости (движения мышью, нажатия клавиш), тем самым приостанавливая взаимодействие ИИ и Хобота.

- **Расширение браузера**  
  - Запускает Хобота **один раз при полной загрузке страницы** поддерживаемого сайта.
  - Передаёт директивы в стандартный вход Хобота через **постоянный канал связи** (chrome.runtime.connectNative).
  - Принудительно **замораживает заголовок вкладки**, добавляя уникальный суффикс для стабильной идентификации окна.
  - Принимает служебные сообщения Хобота через выходной поток в реальном времени.
  - При старте Хобота передаёт ему пакет контекста (сайт, имя окна, уникализированное имя вкладки).

- **Агент Хобот (CLI‑приложение)**  
  - Получает директивы из постоянного входного потока (stdin).
  - Выполняет команды директивы строго последовательно.
  - Формирует отчёт по результатам каждой директивы и вводит его в поле ввода веб‑страницы ИИ путём эмуляции действий пользователя, используя замороженное имя окна для фокусировки.
  - Логирует ошибки и служебные события.  
  - Обнаруживает вмешательство пользователя и прерывает работу по соответствующим правилам.

## 3. Жизненный цикл агента

### 3.1. Запуск и завершение

- Хобот запускается расширением браузера **один раз при загрузке каждой вкладки/сессии** ИИ.
- При старте Хобот получает от расширения вводную информацию:  
  - имя сайта, на котором работает ИИ;  
  - имя окна;  
  - уникализированное имя вкладки (замороженный заголовок + уникальный суффикс).
- Хобот живёт, пока жива соответствующая вкладка/расширение или пока не завершит работу по одной из причин:  
  - таймаут молчания;  
  - фатальная ошибка;  
  - явное завершение по протоколу (если такая возможность будет добавлена позднее).  

### 3.2. Таймаут молчания

- Период молчания задаётся в конфигурационном файле Хобота.  
- Молчанием считается отсутствие данных во входном потоке (stdin) в течение заданного периода.  
- По истечении таймаута Хобот корректно завершает работу.  
- После завершения по таймауту расширение может перезапустить Хобота при возобновлении работы с ИИ.

### 3.3. Перезапуск

- Если Хобот завершился с ошибкой, расширение предпринимает **ровно одну попытку автоматического перезапуска**.
- При неудачном повторном запуске расширение выводит системный алерт пользователю и прекращает работу.
- При новом запуске Хобот не обязан помнить состояние предыдущего процесса; восстановление контекста, если оно требуется, лежит на ИИ.

## 4. Каналы взаимодействия

### 4.1. Входной поток директив (stdin)

- Используется постоянное соединение, которое не разрывается между директивами.
- Структура: текст, содержащий директивы ИИ, оформленные по протоколу с явными маркерами начала и конца (например, `<<<ai N ... >>>ai N`).  
- stdin трактуется как **упорядоченная очередь директив**, Хобот обрабатывает их в порядке поступления.

### 4.2. Служебный выходной поток

- Отдельный служебный поток (stdout) используется для сообщений расширению в реальном времени:
  - Хобот посылает расширению сигнал о том что работа над директивой ИИ или директивой расширения закончена и он готов принять следующую директиву.
  - На основе этого сигнала расширение поддерживает флаг занятости Хобота.
  - критические ошибки с указанием контекста (файл, строка, тип ошибки);  
  - важные служебные события (например, инициализация завершена, не удалось найти окно, и т.п.).  
- Формат сообщений должен позволять расширению однозначно различать тип события и реагировать (например, перезапуск агента при фатальной ошибке).

### 4.3. Отчёты в сторону ИИ

- Все отчёты Хобота для ИИ вводятся в поле ввода веб‑страницы ИИ путём эмуляции действий пользователя (клавиатура/мышь).  
- Для каждой директивы формируется ровно **один** отчёт, оформленный по протоколу ИИ ↔ Хобот (например, `<<<hbt N ... >>>hbt N`, где `N` — номер директивы).  
- В отчёт включается:  
  - результат выполнения каждой команды;  
  - информация о возникших ошибках;  
  - в случае прерывания — указание причины (ошибка формата, вмешательство пользователя и т.п.).

## 5. Работа с директивами

### 5.1. Идентификация и стабильность

- Расширение блокирует любые изменения document.title со стороны сайта, принудительно возвращая уникальное значение с суффиксом.
- Хобот использует этот замороженный заголовок как единственный ключ для поиска окна через WinAPI.


### 5.2. Нумерация и уникальность

- Директива ИИ состоит из открывающего тега, содержимого в формате JSON и закрывающего тега.
- Расширение занимается приемом, проверкой последовательности директив и их отправкой Хоботу.
- Расширение не занимается анализом содержимого директивы, она работает только с ее открывающим и закрывающим тегами.
- Каждая директива имеет уникальный **номер**, который строго возрастает.  
- Расширение ведёт учёт уже **взятых в работу** директив.  
- Повторная директива с номером, который уже был успешно исполнен, **молча игнорируется** и не приводит к повторному выполнению команд.  
- Открывающий и закрывающий маркеры директивы должны содержать совпадающий номер; несоответствие номера закрывающей скобки директивы или ее отсутствие в течение таймаута считается ошибкой.

### 5.3. Строгая последовательность директив.

- В каждый момент времени Хобот **исполняет не более одной директивы**. ИИ не должен подавать следующую директиву до завершения обработки текущей.
- Расширение считает любую новую директиву ошибочной если она приходит в момент, когда флаг занятости Хобота поднят.

### 5.4 Обработка ошибок, выявленных расширением
- Ошибки, выявленные расширением, оформляются директивами расширения и посылаются агенту, поскольку расширение не может посылать сообщения ИИ. Агент переправляет их ИИ.

### 5.5. Строгая последовательность выполнения команд Хоботом

- Внутри директивы команды выполняются **строго последовательно** в порядке их описания.  
- Параллельное выполнение команд внутри директивы Хоботом не допускается.  
- Логика ветвлений, циклов, принятия решений — полностью на стороне ИИ; Хобот реализует только проверку корректности и исполнение команд.

## 6. Обработка ошибок и невалидного ввода Хоботом

### 6.1. Невалидные команды

- Если команда в директиве невалидна (ошибка синтаксиса, несоответствие схеме, невозможность выполнить и т.п.), Хобот:  
  - прекращает исполнение текущей директивы;  
  - формирует отчёт, содержащий:  
    - результаты уже выполненных команд;  
    - описание ошибки на проблемной команде; 
    - копию сообщения помещает в свой журнал ошибок;
  - **очищает текущий входной буфер**: все данные, которые уже приняты, но ещё не разобраны (исключая директивы расширения), отбрасываются.  
- После очистки stdin дальнейшая работа возможна только за счёт того, что ИИ снова начнёт подавать новые директивы с учётом полученного отчёта об ошибке.

### 6.2. Невалидные директивы

- Если нарушен формат начального/конечного маркера директивы или номера в маркерах не совпадают, директива считается невалидной.  
- Хобот формирует отчёт об ошибке протокола для данной директивы и не выполняет ни одной её команды.
- Копия сообщения помещается в журнал ошибок.

### 6.3. Частично валидная директива

- При частично валидной директиве (несколько первых команд корректны, одна из последующих — нет) Хобот:  
  - корректно выполняет команды до первой невалидной;  

### 6.4. Критические внутренние ошибки

- При критической внутренней ошибке (например, фатальная ошибка среды, невозможность продолжения работы) Хобот:  
  - логирует ошибку в журнале ошибки с указанием файла, номера строки, типа ошибки;  
  - отправляет служебное сообщение расширению через stdout с теми же данными;  
  - по возможности (если уже инициализировано нужное окно/вкладка) формирует для ИИ сообщение об ошибке в поле ввода;  
  - завершает работу с ненулевым кодом.  
- Расширение, получив уведомление, перезапускает Хобота по своей логике.

## 7. Вмешательство пользователя

### 7.1. Детекция активности

- Хобот отслеживает активность пользователя на уровне системы:  
  - движения мыши;  
  - нажатия клавиш на физической клавиатуре.  
- Активность считается вмешательством пользователя вне зависимости от того, в каком окне она произошла.

### 7.2. Поведение при вмешательстве

- После **каждой успешно выполненной команды** Хобот проверяет, была ли за время исполнения команды активность пользователя.  
- Если активность обнаружена:  
  - Хобот прекращает дальнейшее исполнение текущей директивы;  
  - формирует отчёт по директиве, включающий результаты уже выполненных команд и указание на прерывание из‑за вмешательства пользователя;  
  - отправляет ИИ отдельный запрос/сообщение о необходимости приостановить диалог (формат определяется протоколом).  
- После этого ИИ должен прекратить отправку новых директив до принятия решения пользователем.

### 7.3. Нет отката действий

- Хобот **не выполняет откат** уже произведённых действий (отмена файловых операций, закрытие запущенных программ и т.п.).  
- Ответственность за возможный откат или компенсацию последствий лежит на ИИ, который принимает решения на основе отчёта Хобота и собственных знаний о сценарии.

## 8. Требования к окружению и правам

- Целевая платформа на текущем этапе — **Windows 10**.  
- Хобот запускается как CLI‑приложение с максимальными доступными правами:  
  - управление окнами и фокусом;  
  - эмуляция клавиатурного и мышиного ввода;  
  - создание скриншотов;  
  - доступ к файловой системе;  
  - запуск внешних процессов и скриптов;  
  - другие действия, необходимые для выполнения команд, определённых протоколом.  
- Вопросы ограничений прав и песочницы считаются вне рамок данного ТЗ и могут быть рассмотрены отдельно в будущем.

## 9. Логирование

- Хобот ведёт журнал ошибок и, при необходимости, ключевых действий:  
  - время события;  
  - тип события (ошибка, предупреждение, служебное сообщение);  
  - контекст (файл, строка, описание команды или директивы);  
  - при критических ошибках — полная диагностическая информация, необходимая программисту.  
- Формат и место хранения журнала могут быть уточнены отдельно и не регламентируются данным документом.
