## Порядок работы (задание):

1. Прочитай структуру каталога `alta`.

2. Выйди в режим диалога, запроси у оператора 
   - Название кейса и номер этапа, 
   - сайт `ai_url` в приглашении Хобота.

3. Прочитай промпты, для текущего этапа.

4. Начинай работу по этапу.

### Примечание
- Перед чтением файлов распечатай состав каталога с их размерами.
- Для загрузки docx, xlsx, pdf, png используй перетаскивание файлов командами Хобота. При превышении объема 15 Мб реши
  вопрос возможности перетаскивания с оператором. Файлы можно перетаскивать пачками.
- xml файлы в кодировке cp1251.
- Для доступа к текстовым файлам используй команды Хобота `read_file`, `write_file`.
- При загрузке пачками текстовых файлов смотри чтобы суммарный размер не превышал 120 кбайт.
- Разрешено читать любые документы в каталоге `alta` и подкаталогах.
- Не спрашивай подтверждений на чтение файлов. 
- Не спрашивай подтверждений на действия, прописанные в промптах.

## 0. Этапы работы

## Этап 0.  Конвертация первички (Опционально)

AI подготавливает первичные документы к обработке: приводит docx, xlsx, pdf и образы к текстовому виду (md).
Если этап не выполнен, то этап 1.0 будет брать данные прямо из первички.

После завершения этапа напомнить оператору очистить историю / пересоздать чат.

Исходные данные:
- первичные документы `alta\source\<имя кейса>\...` 
- промпт `alta\prompt\stage_0\doc_conversion_schema.md`

Результаты `alta\source\<имя кейса>\md\...`
- `*.md` текстовые версии документов
- `doc_conversion_review.md` 

## Этап 1.0  Сбор и нормализация фактов
- Если этап 0 был выполнен, используй `alta\source\<имя кейса>\md\...`. 
- Если md версий документов не существует или они содержат недостаточно данных, уведомить оператора и использовать первичку.

**Результаты в `alta\stage_1.0_result\<имя кейса>\`:**
- `primary.md` — нормализованная база фактов поставки (источник истины).
- `primary_review.md` — отчет по этапу, вопросы к оператору.

**Исходные данные:**
- `alta\source\<имя кейса>\...` (первичка и/или md-версии из Stage 0)
- `alta\stable_source` (не меняются между поставками; xml, cp1251)

**Промпты:** `alta\prompt\stage_1.0\primary_schema.md`, `primary_review_schema.md`.

**Результаты в `alta\stage_1.0_result\<имя кейса>\`:**
- `primary.md` — нормализованная база фактов поставки (источник истины).
- `primary_review.md` — отчет по этапу, вопросы к оператору.

## Этап 1.1. Формализация документов (XML)

Генерация пакета XML-файлов на основе собранных фактов для импорта в Альту.

**Исходные данные:**
- `alta\stage_1.0_result\<имя кейса>\primary.md`
- Первичные документы (только для подстановки полных текстов/base64 по ссылкам).

**Промпты:** `alta\prompt\stage_1.1\doc_xml_schema.md`, `doc_xml_review_schema.md`.

**Результаты в `alta\stage_1.1_result\<имя кейса>\`:**
- `formalized_docs\*.xml` — файлы для импорта в Альту.
- `doc_xml_review.md` — отчет по генерации.


## Этап 2.0. Подготовка полей ДТ

Проекция фактов из `primary.md` на структуру полей декларации (графы ДТ).

**Исходные данные:** `alta\stage_1.0_result\<имя кейса>\primary.md`.

**Промпты:** `alta\prompt\stage_2.0\dt_fields_schema.md`, `dt_fields_review_schema.md`.

**Результаты в `alta\stage_2.0_result\<имя кейса>\`:**
- `dt_fields.md` — данные, подготовленные для генерации XML ДТ (источник истины для этапа 2.1).
- (в полном режиме) `dt_fields.md` — представление `dt_fields.md` в виде Markdown-таблиц для оператора.
- `dt_fields_review.md` — отчет по этапу.

## Этап 2.1. Генерация XML ДТ

Формирование финального XML файла декларации.

**Исходные данные:** `alta\stage_2.0_result\<имя кейса>\dt_fields.md`.

**Промпты:** `alta\prompt\stage_2.1\dt_xml_schema.md`, `dt_xml_review_schema.md`.

**Результаты в `alta\stage_2.1_result\<имя кейса>\`:**
- `dt_xml.md` (или .xml) — готовая декларация для импорта.
- `dt_xml_review.md` — финальный отчет.

## 1. Базовые правила

### 1.1. Никакой отсебятины

Любое значение, попадающее в `primary.md`, должно быть:
- извлечено из первичных документов;
- либо подтверждено оператором;
- либо корректно выведено из подтвержденных данных;
- либо явно помечено как эталонное наблюдение, не являющееся фактом новой поставки.
  Если данных не хватает — остановиться и обратиться к оператору либо явно зафиксировать пробел.
