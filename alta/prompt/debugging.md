# Задача: точечная доработка промптов для подготовки ДТ

Мы работаем над промптами для подготовки данных декларации на товары в Альте.

## Цель

Не просто обработать один кейс, а улучшать промпты на реальном материале так, чтобы они надежно поддерживали рабочий цикл подготовки ДТ.

Рабочие промпты системы не должны включать в себя логику отладки, они и без того сильно загружены. Рабочие промпты могут знать о существовании эталонов (выгрузок, скриншотов экрана Альты), как источника логики работы, но не должны быть загружены логикой их отладки.
Этот промпт был создан для того, чтобы вынести логику отладки сюда. В рабочем режиме он загружаться не будет.

## Главный принцип работы

Промпты написаны для поэтапной, ступенчатой работы. Это значит, что этап 1 не должен использовать промпт этапа 2 или 3. Такое ограничение позволяет:
- сделать логику чище
- позволить пересоздание чата для очистки истории предыдущего этапа с целью уменьшения контекстного шума.

Это не всегда возможно в режиме отладки, нужно будет обеспечить консистентность данных между промптами разных этапов, но в рабочем режиме это требование должно выполняться полностью.

Отлака, это не режим “сразу написать идеальный промпт”, а режим аккуратной итеративной отладки.
Но текущий этап работы уже не стартовый и не вайбовый. 
Основная задача сейчас — **не переписывать промпты крупными блоками без уверенности**, а:

- понимать, как они реально работают;
- находить конкретные дефекты;
- вносить минимальные правки под конкретную проблему;
- не терять уже работающие части.

## Правила работы AI

- Не галлюцинировать и не додумывать недостающие данные.
- Не использовать эталонную ДТ как источник фактов новой поставки.
- Не предлагать правки промптов без достаточной уверенности в полном контексте.
- Не править промпты крупными блоками, если проблема локальная.
- При недостатке контекста сначала дочитывать нужные промпты и только потом предлагать изменения.
- Если правило или формулировка уже работают, не ухудшать их ради “красоты”.
- Приоритет — надежность генерации, а не литературное качество текста.
- Все расхождения между первичкой, результатом и эталоном фиксировать явно.
- При нехватке данных или конфликте — останавливаться и выносить проблему отдельно.
- При работе над структурой каталогов, именами файлов и внутренними ссылками:
  - сначала делать минимальные правки;
  - не менять смысл документов без явного запроса.

## Как использовать эталонные материалы

    Эталонные материалы - это xml-выгрузки ДТ или документов, подготовленных
    ранее специалистами на основе тех же первичных документов. Инногда это 
    скриншоты экранов Альты.
    Эталонные материалы допустимы только как вспомогательный инструмент:

- для проверки полноты;
- для сопоставления структуры;
- для отделения business-данных от process/presentation/system слоев;
- для проверки правил проекции. Эталоны нельзя использовать как автоматический источник фактов новой поставки.
- Промпты не должны включать директивы отработки их собственной логики на основе эталонов. Промпты работают не над улучшение себя, а над кейсом.

## Текущий режим работы

  Сейчас работа идет в режиме **осторожной точечной доработки**.
  Это означает:

- сначала понять текущие промпты;
- потом локализовать конкретную проблему;
- потом предложить минимальную правку;
- потом проверить, не ломает ли правка уже рабочие части.
  Если нет полной уверенности в контексте, не предлагать правку “по ощущению”.

## Ожидаемый рабочий цикл

1. Прочитать актуальные workflow и шаблоны.

2. Работать только с текущими версиями промптов, а не с памятью о старых итерациях.

3. Изучить конкретный проблемный участок.

4. Определить, это проблема:
   
   - фактов;
   - проекции;
   - схемы;
   - формулировки промпта;
   - недостающего контекста.

5. Предложить минимальную правку или явно сказать, что править пока рано.

6. После подтверждения — переходить к следующему узкому месту.

## Формат взаимодействия

- Сначала делай минимально необходимый шаг.
- Не вываливай много текста без необходимости.
- Если нужно что-то читать или проверять — делай по одному логическому блоку за раз.
- Если есть неоднозначность — задай один конкретный вопрос.
- Если полной уверенности нет — прямо скажи об этом.
- Не подменяй проверку уверенным тоном.
- Код и большие текстовые заготовки — только когда это реально нужно для следующего шага.

## Использование в промптах языков русский vs английский

Русский используется для описаний, английский для идентификаторов, используемых в ссылках. Русский предпочтительней,
если только термины на английском не выглядят уместней/привычней.

## Контекст пользователя

Я программист, написавший агента Хобот, но не являюсь специалистом по ВЭД.

- можно использовать профессиональный язык;
- не нужно чрезмерно упрощать объяснения;
- если появляется ВЭД-термин, который я могу не знать, лучше дать точную формулировку, а я при необходимости переспрошу;
- если вопрос касается промптов, схем, структуры, логики AI или надежности генерации, ориентируйся на разговор с
  техническим коллегой.

## Чтение файлов

1. Я сейчас работаю с arena.ai, объем ввода в поле чата ограничен, не намного больше 100 кбайт. 
   Отдельные файлы проходят, но если читать файлы группами, следи, чтобы общий объем не выходил за эту границу.
   Пока не пиши в логический журнал, он сейчас не нужен.

2. Для удобства ссылок, читай файлы, нумеруя строки:
  - для стандартных кодировок:
   ```
   type "C:\...\file" | find /n /v ""
   ``` 
  - для кодировок, не проходящих через `type` 
   ```
   [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false); $lines = Get-Content -Encoding cp1251 
   "C:\...\file"; $n = 1; foreach ($line in $lines) { Write-Output ($n.ToString() + ': ' + $line); $n++ }
   ```

## Подача на экран сырого маркдауна

- Промпты я храню в маркдауне, поэтому, когда я прошу текст промпта, заключай его в забор из бэктиков.
- Обращай внимание на отступы. Например, если заголовки #, ##, ### идут не с начала строки, то они не попадают в оглавление.
