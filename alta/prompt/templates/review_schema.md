# review_schema

## Назначение

`review.md` — это краткий, понятный и практический отчет по состоянию кейса.
Он нужен, чтобы оператор быстро понял:

1. какие документы разобраны;
2. что подтверждено по поставке;
3. чего не хватает;
4. какие проблемы относятся к:
   - первичке,
   - master data Альты,
   - расчетам,
   - mapping / process rules;
5. можно ли переходить к:
   - `ui_input.md`
   - `xml_import.md`

---

## Общий принцип

`review.md` — это **не хранилище всех данных**, а **сводка состояния**:

- что уже надежно известно;

- что пока не подтверждено;

- что блокирует следующий этап;

- какие process / mapping rules уже нужны для следующего шага.
  
  ### Практический лимит detail-level

- `review.md` не должен превращаться во вторую копию `facts.md`.
  Если деталь:

- не влияет на следующий шаг;

- не объясняет блокер;

- не влияет на построение `ui_input.md` или `xml_import.md`;
  то ей не место в `review.md`.

## 9. Что делать нельзя

- не использовать эталонную ДТ как источник истины для новой поставки;
- не переносить автоматически reference-only поля в shipment layer;
- не записывать process / mapping rules как document-confirmed facts поставки;
- не строить финальный `xml_import.md`, пока не закрыты критичные пробелы по shipment data / master data / calculations / mapping rules;
- не смешивать данные поставки, master data, расчетные поля и mapping rules в один неразмеченный слой;
- не считать, что совпадение с прежними кейсами автоматически делает значение shipment fact.
