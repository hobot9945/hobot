# Шаблон отчета по этапу 1.0: Сбор и нормализация первичных данных (primary_review_schema.md)

## Назначение
Этот файл является сводным отчетом по результатам разбора первичных документов.
Его главная цель — предоставить оператору понятную картину того, что удалось извлечь, где найдены противоречия
(конфликты) и каких данных не хватает (`pending`).

AI формирует `primary_review.md` на основе `primary.md` (источник истины этапа 1.0).

## Требования к оформлению
- Вопросы предназначены для чтения человеком: должен быть указан файл-источник и место внутри документа (страница/раздел/
  строка таблицы/фрагмент текста).
- Если вопрос касается значения поля, идентификатора поля недостаточно: должно быть описано назначение поля.
- Писать детерминированно: не добавлять предположений и “улучшений” данных, которых нет в `primary.md`.

## Структура файла `primary_review.md`

### 1. Метаданные и статус
- **Кейс:** `<Название кейса>`
- **Статус готовности:** `<Ready / Blocked / Partial>`
- **Всего обработано документов:** `<число>`
- **Всего сформировано полей:** `<число>`
- **Количество конфликтов:** `<число>`
- **Количество недостающих данных (Pending):** `<число>`

### 2. Использованные документы
Формат: по каждому документу — отдельный блок:

### <DocumentType>

- <uqi_prefix>
- <relative_path_to_file> (относительно папки поставки `...\md\` или `alta\stable_source\`)
- <status>                (`OK` | `Pending (<N>)`)

Пример:

### Contract
- formalized.contract_1
- md\SALES CONTRACT NoLM-2553.md
- OK

### Service Invoice
- formalized.service_invoice_1
- md\Счет_№26-00378-tl_от_27-01-2026.md
- Pending (7)

Правила:
- `<relative_path_to_file>` — путь **как на диске**, но без абсолютного префикса.
- Если документ использует несколько файлов (например master data), перечислять несколько строк `- <path>` внутри одного блока.
- Если документ найден в md-папке, но НЕ использован (не сматчился ни с одним шаблоном) — не включать сюда; вынести в раздел 4.

### 4. Существенные данные первички, которые не попали в `primary.md`
Если по каким-либо причинам данные не удалось поместить в `primary.md`, они фиксируются здесь с указанием причины.
Например:
- нет подходящего шаблона для документа / поля;
- данные есть, но неоднозначны и требуют решения оператора;
- данные слишком объемные — перенесены как `link` (указать куда).

Существенными считаются данные, которые необходимы для формирования ДТ или импортируемых формализованных документов.

Формат (на каждый пункт):
- Источник: `<имя файла>`
- Данные: `<что именно>`
- Причина: `<почему не попали>`
- Влияние: `<что блокирует / где понадобится дальше>`

### 5. Выявленные конфликты (ТРЕБУЕТСЯ РЕШЕНИЕ)
В этот раздел попадают все расхождения между документами (например, разный вес в CMR и Инвойсе, разные условия
поставки и т.д.).

Формат для каждого конфликта:
- **Конфликт #<N>**: `<Краткое описание, например: Расхождение в весе брутто>`
  - Поле (UQI): `<целевой UQI>`
  - Документ 1 (`<имя файла>`): `<значение 1>`
  - Документ 2 (`<имя файла>`): `<значение 2>`
  - **Вопрос оператору:** Какое значение использовать для формирования ДТ/этапа 1.1?

### 6. Недостающие данные / Pending (ТРЕБУЕТСЯ ОТВЕТ)
В этот раздел попадают все поля, значения которых отсутствуют в документах, не читаются, или их вывод невозможен
без догадок.

Формат для каждого вопроса:
- **Вопрос #<N>**: `<Что нужно узнать>`
  - Документ/Сущность: `<тип документа или сущности>`
  - Поле (UQI): `<UQI, который получил статус pending>`
  - Контекст: `<файл-источник + место внутри документа>`
  - **Вопрос оператору:** `<прямой вопрос>`
